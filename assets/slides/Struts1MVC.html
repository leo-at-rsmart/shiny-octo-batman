Struts 1 and Model View Controller

Struts 1 based-on Model View Controller Enterprise Design Pattern

Creates a separation of concerns by splitting software into roles of:

  • Model
  • View
  • Controller

JSP/JSTL is considered the View

  • Provides templating for the view and access to the model through struts tags
  • Rice and KFS manage templating and modularity through custom tag files.

The Model is represented in several different ways.

  • Struts provides a layer between the view and the Model which has two parts (Action and Action Form)
  • The Model is actually separate from Struts 1. These are the Documents and Business Objects.

Action and ActionForm

  • Provide a midway point for the Controller to pass the Model to the View without needing to know details about either.
  • ActionForm is basically a POJO wrapper around the Model. Represents the HTML Form.
  • Action handles mapping a view to an ActionForm.

work/web-root/WEB-INF/struts-config.xml

Handles ActionForm definitions and makes Actions aware of them by mapping them.

Controller is called KualiRequestProcessor.

  • Rarely needs customization

Struts 1 is KNS only. KRAD Uses Spring MVC and Tiles.

Important mostly for Transactional Documents built with KNS.

Accessing Spring Services from Struts

Can be done using the Struts Action.

  • Invoked per/request
  • Best practice is to add as much functionality to Spring services instead of the Struts Action
